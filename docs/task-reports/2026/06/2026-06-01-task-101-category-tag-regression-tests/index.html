Grimoire Task Report

Category And Tag Regression Tests

June 1, 2026 TASK-101 Done

Before

Category and tag work from TASK-095 through TASK-100 had strong feature tests, but the cross-task regression surface was spread out. One category reparenting edge case only checked the target parent depth, not the depth of the moved category's descendants.

Problem

TASK-101 needed durable coverage for empty rows, aggregate counts, delete/rename/move behavior, and filter refresh. Reparenting a branch with existing descendants also needed to preserve the documented three-level category limit.

Changes Added

  • Added daemon regression tests for empty tags, active-only tag counts, category active counts, category delete cleanup, and tag delete filter/search cleanup.
  • Added a category repository subtreeHeight helper and route validation so branch moves cannot push descendants beyond the maximum depth.
  • Added frontend coverage for selected category rename/delete filter refresh, category move dialog behavior, bookmark-hook aggregate refetches, and tag-detail cache invalidation.
  • Corrected parity documentation so TASK-100 and TASK-101 statuses match the task board.
  • Moved TASK-101 to done with verification notes and linked task-report evidence.

Result

Category and tag parity behavior now has a focused regression suite. The daemon rejects invalid whole-branch reparent operations with 422, while existing empty-row, count, delete, rename, and frontend refresh paths are pinned down by tests.

Implementation Notes

The new depth validation computes the target depth of the moved node plus the deepest descendant distance. This keeps valid leaf and shallow branch moves working while rejecting moves that would produce a fourth visible category level.
