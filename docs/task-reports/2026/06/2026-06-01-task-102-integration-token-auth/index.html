Grimoire Task Report

Integration Token Authentication

2026-06-01 TASK-102 Completed

Before

The daemon had loopback and origin hardening, but MCP and local API clients had no managed bearer-token path.

Problem

Grimoire parity needed pragmatic local integration auth without changing the trusted first-party browser flow or implying public-server support.

Changes Added

  • Added a SQLite-backed integration_tokens store with hashed token values, redacted prefixes, last-used tracking, rotation, and revocation.
  • Added /integration-tokens management routes and protected /mcp with bearer-token middleware.
  • Kept regular first-party REST routes tokenless, while validating any presented bearer token on those routes.
  • Rejected oversized and non-JSON token-create requests before they can be silently accepted.
  • Regenerated API docs and updated security, MCP, Docker, overview, parity, and release-decision documentation.

Result

Local MCP clients now need a managed integration token. Invalid, rotated, and revoked tokens consistently return 401 problem responses with a bearer challenge, while loopback browser usage remains unchanged.

Verification

  • bun test src/test/integration/integration-token-auth.test.ts passed after the RED failure.
  • bun test src/test/integration/integration-token-auth.test.ts src/test/migrations.test.ts passed.
  • npm run docs:api:check passed after regenerating API.md and docs/api-contract.json.
  • npm run check passed with existing lint/build warnings only.
Implementation Notes

The auth layer is intentionally scoped. /mcp requires tokens because it is an explicit integration surface. Regular REST routes remain accessible to the first-party app without tokens, but a supplied Authorization header must verify against a non-revoked integration token.
