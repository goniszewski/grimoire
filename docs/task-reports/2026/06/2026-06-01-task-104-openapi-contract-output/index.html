Little Imp Task Report

OpenAPI Contract Output

2026-06-01 TASK-104 Completed

Before

The daemon contract generated API.md and docs/api-contract.json, but local client tooling that expects OpenAPI had no generated artifact to consume.

Problem

Grimoire parity calls for OpenAPI-compatible output from the same source contract, including route parameters, request bodies, responses, error shapes, pagination schemas, and managed integration-token auth.

Changes Added

  • Added docs/openapi.json generation from daemon/src/api/contract.ts.
  • Mapped daemon paths, query/path parameters, request bodies, response content, schemas, tags, server metadata, and MCP bearer-token security into OpenAPI 3.0.
  • Added npm run docs:openapi and npm run docs:openapi:check, while keeping npm run docs:api:check responsible for all generated API artifacts.
  • Documented OpenAPI consumption and limitations in generated API.md.

Result

Local integration authors can now point OpenAPI-aware tooling at docs/openapi.json without hand-maintained schema translation, and drift checks fail if the generated artifact falls behind the daemon-owned contract.

Verification

  • npx vitest run scripts/api-docs-generator.test.ts failed RED for missing OpenAPI output and invalid empty required arrays, then passed with 9/9 tests.
  • npm run docs:api regenerated API.md, docs/api-contract.json, and docs/openapi.json.
  • npm run docs:api:check and npm run docs:openapi:check passed after regeneration.
  • npm run check passed, including lint, type-checks, frontend tests, daemon tests, API docs drift, and production build. Existing Fast Refresh, React Router future-flag, Browserslist, and chunk-size warnings remain.
  • git diff --check passed.
Implementation Notes

The OpenAPI generator keeps Hono ALL routes compatible by emitting the primary client method POST and preserving the source method in x-little-imp-source-method. Mixed content responses keep the documented media types, while CSV/media/SSE clients still need content-type-specific handling.
