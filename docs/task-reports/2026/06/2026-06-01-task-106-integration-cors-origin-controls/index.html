Grimoire Task Report

Integration CORS And Origin Controls

2026-06-01 TASK-106 Completed

Before

Little Imp already kept unsafe browser writes behind a loopback origin allowlist, but rejected CORS preflights fell through to a generic no-reflection response and the generated API docs did not explain local client origin setup.

Problem

Grimoire parity calls for conservative local integration CORS controls with deliberate trusted-origin configuration, clear failure behavior, and documentation that keeps browser-capture flows out of the current batch.

Changes Added

  • Added explicit 403 responses for rejected CORS preflight requests before the CORS middleware can emit a generic 204.
  • Added daemon coverage for no-Origin local clients, allowed loopback preflights, unsafe preflight rejection, and configured loopback origins.
  • Documented local CORS_ORIGINS setup in generated API.md with loopback-only constraints.
  • Updated the security boundary with the implemented TASK-106 origin policy and preflight expectations.

Result

Local browser clients can be configured deliberately with loopback origins, local scripts continue to work without CORS headers, and unsafe public browser origins receive predictable rejection for writes and preflights.

Implementation Notes

The policy remains intentionally narrow. Missing Origin is allowed for non-browser local clients but does not reflect CORS. Configured origins are normalized and accepted only when they are loopback HTTP or HTTPS origins. Non-loopback entries stay ignored even if present in CORS_ORIGINS.
