Security Hardening Review Fixes

Closed the high and medium findings from the comprehensive code review: SSRF redirect handling, update-check source abuse, upgrade authenticity, bind-address policy, URL credential hygiene, body limits, restore checksum bypass removal, and frontend mutation/open safety.

SecurityDaemonFrontendAPI

Problem

A review of the most critical paths found several security and correctness gaps that still mattered under the local-first threat model: page fetch followed redirects before validating private targets, GET /updates/check?source= accepted arbitrary URLs, upgrades could install with checksum-only trust, Docker/native HOST=0.0.0.0 was easy to misuse, and the UI often toasted success before the daemon confirmed mutations.

Changes

  • Manual redirect validation for page fetch and update checks; HTTP update checks reject arbitrary source query params.
  • Remote upgrades require HTTPS release bases, detached signatures, optional fingerprint pinning, and download size caps; --allow-unsigned is an explicit escape hatch.
  • Bind-host policy refuses non-loopback unless container/opt-in is present, with a loud warning.
  • Shared public URL validation rejects credentials across bookmarks, capture, MCP, and import; open/href sinks check http(s) safety.
  • Default body limits on mutating routes; HTTP restore no longer accepts allow_unsafe_no_checksum.
  • Add/delete await daemon confirmation before success toasts; category bulk move uses IDs; React error boundary; bookmarklet uses JSON.stringify.

Verification

  • Focused daemon tests for fetch redirects, public URL parsing, bind-host policy, updates route, and CLI upgrade signature requirements.
  • npm run docs:api regenerated contract/OpenAPI/API.md after restore/import contract changes.
  • Broader npm run check / daemon / e2e verification in the PR CI loop.
